Why it matters

Most security workflows fail when the claim gets stronger than the evidence.

Hunter is built to reduce that gap. It pushes security work toward proof, comparability, and bounded language instead of inflated conclusions.

1. Understand the protocol

Map roles, transitions, control surfaces, and assumptions before making stronger exploit claims.

2. Validate real behavior

Use the target repository's own execution surface wherever possible instead of a disconnected demo environment.

3. Compare outcomes

Separate baseline, control, and exploit behavior so the difference is measurable instead of rhetorical.

4. Keep the claim bounded

Preserve downgrade logic, evidence trails, and explicit caveats so decision-makers know what is proven and what still needs review.

Case studies

Two examples of how Hunter turns signals into reviewable evidence.

Phoenix Bonds shows repo-native failure-shape validation. Whitechain shows structured semantic exploit confirmation with bounded severity language.

Phoenix Bonds Repo-native validation

From exploit-family prioritization to reproducible failure evidence

Hunter was used to move from a suspicious underwater-settlement scenario into repo-native evidence that reproduced the failure shape inside the target workflow.

What Hunter did

  • Prioritized the exploit family
  • Lifted the target repo's own harness
  • Generated a runnable two-user scenario
  • Validated the behavior through controlled reruns

Observed result

  • One user exited first after adverse movement
  • Follow-on cancel path failed
  • Commit path failed independently
  • The remaining user path became operationally stuck
Whitechain Semantic exploit confirmed

From static signature-binding signal to structured exploit confirmation

Hunter was used to validate a bridge signature-binding issue through repo-native execution, structured semantic markers, and within-run comparison of signed and executed identities.

What Hunter proved

  • The signed mapping and executed mapping diverged
  • The signed origin token and selected origin token differed
  • The relevant signed hash still validated
  • The result was preserved as first-class evidence artifacts

How it was framed

  • Repo-native test execution in the target repository
  • Structured semantic marker extraction
  • Machine verdict: semantic exploit confirmed
  • Severity kept bounded as a critical candidate pending manual review
